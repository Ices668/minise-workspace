# OpenClaw 使用经验与技巧分享 - 大纲

## 主题定位
- 标题：把 OpenClaw 用成个人工作台：我的实战经验、工作流和技巧
- 场景：公司内部分享
- 目标：让同事理解 OpenClaw 不只是聊天工具，而是“消息入口 + 工具编排 + 技能体系 + 工作流沉淀”的个人操作系统

## 页结构

### 1. 封面
- 标题
- 副标题：从“会聊天”到“真能干活”
- 演讲者 / 日期

### 2. 为什么值得用 OpenClaw
- 我们日常的典型痛点
- 工具太散、上下文断裂、重复劳动高
- OpenClaw 的价值：把消息、工具、记忆、自动化串起来

### 3. 我理解的 OpenClaw = 4 层能力栈
- 消息入口层：飞书/聊天即入口
- 工具执行层：文件、浏览器、shell、文档、设备
- 技能封装层：skill 让复杂能力可复用
- 工作流沉淀层：把一次次成功经验变成固定打法

### 4. 我现在的真实使用方式
- 主通道：飞书直连
- 主工作区：minise-workspace
- 主协作模式：直接对话 + 技能调用 + 文件沉淀
- 关键原则：所有产出统一落盘、可回溯、可复用

### 5. 我的核心工作流（总览）
- 需求进入
- 判断是：信息检索 / 文档处理 / 设计展示 / 编码执行 / 自动化协作
- 选择 skill / 工具
- 产出到 workspace
- 必要时 push / 回传 / 复盘

### 6. 案例 1：产品设计展示工作流
- enhance-prompt：把模糊需求变成高质量 prompt
- stitch-designer：出设计稿
- design-md：沉淀设计系统
- Boss 确认后再进入 stitch-frontend 工程还原
- 价值：把“设计-确认-还原”拆成清晰阶段，减少返工

### 7. 案例 2：工程还原与交付
- stitch-frontend：按设计稿手写还原
- shadcn-ui：组件选型与集成
- github skill / push 脚本：产出自动同步
- 关键铁律：1:1 复刻、图片保留、以设计稿为真相

### 8. 案例 3：知识整理与复杂文档产出
- 用户旅程图 / Road Map 项目经验
- HTML 适合复杂排版展示，PDF 适合最终交付
- 不是每个任务都要“写系统”，很多时候先把信息表达做好
- OpenClaw 适合做“资料理解 + 内容重组 + 交付格式切换”

### 9. CCG 视角：为什么这套方式能持续进化
- CCG 核心思路：让能力不是一次性 prompt，而是可组合、可编排、可沉淀
- 技能像插件，工作流像 SOP，记忆像组织经验库
- 不是单次提效，而是把个人方法论固化

### 10. 我最常用的技能与它们各自负责什么
- enhance-prompt：把需求讲清楚
- stitch-designer：做设计稿
- design-md：抽设计系统
- stitch-frontend：还原工程
- github：同步工作区
- skill-vetter：新 skill 安全检查
- playwright / browser：自动化、截图、抓取

### 11. 真正有用的 8 个实战技巧
- 让所有产出进入统一 workspace
- 先定工作流，再让模型干活
- 复杂任务拆阶段，不要一口气让它“全做完”
- 把可复用经验写进 MEMORY.md / skill / 脚本
- 用 skill 代替重复解释
- 用消息通道驱动任务，而不是来回切工具
- 对外动作谨慎，对内动作果断
- 每次交付后顺手整理一次方法

### 12. 哪些场景最适合上 OpenClaw
- 频繁跨工具
- 需要沉淀经验
- 重复型知识工作
- 有个人工作台 / 团队工作流诉求
- 要求“边做边留痕”

### 13. 哪些坑我踩过
- 工具能用 != 工作流就顺
- 太早工程化，容易返工
- 没有统一产出目录，后面很难找
- 截图/消息/文件格式要提前考虑
- 某些平台能力有限，要提前准备替代方案

### 14. 一句话总结
- OpenClaw 最强的不是回答问题，而是把“我做事的方法”变成可执行系统
- 从助手，到工作台，到个人作战系统
